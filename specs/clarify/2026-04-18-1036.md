# Clarify Session 2026-04-18 10:36

## Idea

在 zenbu-milestones 儀表板上新增「對 zenbuapps org 的 GitHub Issue 進行寫入操作」的能力（V1）。

目前儀表板是純唯讀（每小時由 GitHub Actions 抓資料 → 產靜態 JSON → 部署到 GitHub Pages），
V1 要讓使用者能從 UI 端直接對 issue / milestone 發起變更（例：建立 issue、編輯 issue、留言、關閉、指派 milestone 等），
並同步反映在底下的 zenbuapps/{repo} GitHub Issues 上。

限制：
- 繼續以 GitHub Pages 靜態部署為基礎（無自有後端）
- 必須使用專案現有的 `ZENBU_ORG_READ_TOKEN` 之外的寫入權限 token（權限最小化）
- V1 不做媒體檔上傳（圖片 / 影片）

## Q&A

- Q1: 寫入操作應該如何觸發（走哪條管道把用戶動作送到 GitHub）？
  → A: C — **GitHub Actions + `repository_dispatch`**（前端送 `repository_dispatch` event 觸發一個負責寫入的 workflow，由 workflow 以 write PAT 執行 Octokit 寫操作，避免寫權限 token 落到前端 bundle）

- Q2: 寫入權限的 token 策略？
  → A: B — **職責分離**：新增 `ZENBU_ORG_WRITE_TOKEN`（僅 `issues:write`，fine-grained），保留既有 `ZENBU_ORG_READ_TOKEN`（維持唯讀權限）。寫入 workflow 用 write token，read / rebuild workflow 繼續用 read token。

- Q3: V1 是否支援在 issue 編輯器上傳媒體（圖片 / 影片）？
  → A: A — **V1 不做媒體上傳**；但**必須**在編輯器旁邊加一段教育提示 UI，引導用戶到外部（imgur / YouTube / 另一個 public 的 GitHub Issue）上傳後，把連結貼回來。
     - 教育提示 UI 規格已列入後續規格任務。

- Q4: 寫入管道的最終架構（覆蓋 Q1 的 `repository_dispatch` 決策）？
  → A: **Cloudflare Worker proxy**。前端 → Cloudflare Worker（驗證 Turnstile challenge）→ GitHub REST API。寫入 PAT 存放於 Worker 的環境變數（secrets），絕不出現在前端 bundle。此方案取代 Q1 的 `repository_dispatch` 架構，理由是 Worker proxy 的延遲（< 1s）遠低於 Actions 排隊（可能數十秒），且可直接回傳 GitHub API 的結果（如新建 issue 的 number / URL）給前端即時顯示。

- Q5: Cloudflare Worker 程式碼放哪裡？
  → A: A — **放本 repo 的 `worker/` 子目錄**，與前端共用同一個 `build-and-deploy.yml` workflow 部署（前端 → GitHub Pages；Worker → Cloudflare via Wrangler action）。monorepo 形式，PAT rotation 與前端改版共用一個 PR 流程。

- Q6: 訪客可以做哪些寫入操作？
  → A: D — **訪客可「建立 issue」+「留言任何 issue」**。**關閉 issue 不開放給訪客**，保留給 maintainer 透過 GitHub 原生 UI 操作（避免被惡意濫用關閉他人 issue）。

- Q7: 建立 issue 的表單欄位與限制？
  → A: C — 欄位：
     - `title`：最長 100 字
     - `body`：最長 5000 字
     - `labels`：訪客可選 1 個預設類型 label（`bug` / `feature` / `question`）
     - 後端（Worker）強制附加 `triage-pending` label（無法由前端取消），讓 maintainer 事後分流
  → ⚠ Q14 修訂：分類機制改走 **GitHub Issue Types**（org-level feature），非 label。詳見 Q14。

- Q8-Q10（前序批次）：略（UX / Toast / 樂觀更新策略 — 見先前批次紀錄）

- Q11: 留言成功後 UI 如何同步？
  → A: C — **混合方案**：MSW fixtures 模擬靜態 JSON 讀取路徑 + 寫入成功後樂觀更新本地 state（append 新 comment）；下一小時的 fetcher 重跑時才同步到 `public/data/`。

- Q12: Issue body / 留言的 Markdown 編輯器？
  → A: B — **@uiw/react-md-editor**。支援預覽、工具列、GitHub flavored markdown，與本專案 Tailwind 3 相容。

- Q13: 建立 issue 的 UI 呈現方式？
  → A: A — **Modal 對話框**（max-w-2xl），在 Roadmap 頁面右上角放「建立 issue」按鈕，點擊後彈出表單。留言也走同樣的 Modal 模式（從 IssueList 的 issue 項目點「留言」）。

- Q14: Issue 分類機制（覆蓋 Q7 的 label 方案）？
  → A: 微調版 — **只預建「待審核」label**（給 triage-pending 使用），bug / feature / task 走 **GitHub Issue Types**（org-level feature，非 label）
     - 技術細節：
       - Issue Types 是 2024/09 GA 的 organization-level 功能，需 zenbuapps org owner 在 org settings 啟用並定義 types
       - REST API 支援 2025-03-18 GA：`POST /repos/{owner}/{repo}/issues` 的 request body 加 `type: "Bug"` 欄位（字串值，非 slug）
       - `GET /orgs/{org}/issue-types` listing endpoint 需要 `admin:org` scope → 訪客 Worker 不可直接呼叫，改由 Worker 以寫入 PAT 代查（或 fetcher 在 build-time 抓一次寫到 `public/data/issue-types.json`）
       - @octokit/rest v21.1.1（已安裝）auto-generated from OpenAPI，應已支援 `type` 欄位
     - Label 名稱 clarify 待 Q16 確認（中文「待審核」vs 英文 `triage-pending`）
     - Fetcher 需擴充 `IssueLite` 增加 `type?: string | null` 欄位（data-contract.rule.md 的契約變更，三端同步）

- Q15: 舉報濫用 issue 的聯絡方式？
  → A: B — **單一 contact 欄位 + body template**：在 issue body 提示中放一段預設 `### 舉報聯絡方式\n(選填) Email / GitHub handle：` 的 template，訪客選填即可。不強制、不做單獨欄位。

- Q16: `triage-pending` 這個 label 的最終名稱？
  → A: A — **使用中文「待審核」label**（覆蓋 AI 推薦的英文 `triage-pending`）
     - Label name = `待審核`（UTF-8 中文字）
     - Worker 呼叫 GitHub API 時（例如 `POST /repos/{owner}/{repo}/issues` 的 `labels` 陣列）必須將此字串 **URL-encode** 後才放進 request
     - 該 label 必須由 maintainer 預先在 repo 上建立（或由 Worker 在偵測到不存在時自動 `POST /repos/{owner}/{repo}/labels` 建立一次）
     - 用戶明確選擇此選項，尊重決定

- Q17: Cloudflare Turnstile 的接法？
  → A: B — **Managed mode + Modal 內嵌錯誤訊息 + 環境變數分離**
     - Frontend：Managed widget 直接嵌在建立 issue / 留言 Modal 內（不開新頁）
     - Site Key（公開）→ `VITE_TURNSTILE_SITE_KEY` 環境變數，build 時注入前端 bundle
     - Secret Key（私密）→ Worker 的 wrangler secret（`wrangler secret put TURNSTILE_SECRET_KEY`），前端 bundle **絕不包含**
     - 驗證失敗時：Modal 內 inline 顯示錯誤訊息，不跳出 Modal、不 redirect

- Q18: Worker 的部署域名策略？
  → A: A — **Cloudflare Workers 免費子域 + CORS 白名單**
     - Domain：`<worker-name>.<cf-account>.workers.dev`（例如 `zenbu-milestones-worker.zenbuapps.workers.dev`）
     - 不綁定 custom domain（省下 DNS / SSL 設定成本；免費額度足夠 V1 使用量）
     - CORS `Access-Control-Allow-Origin` 白名單：
       - `https://zenbuapps.github.io`（production）
       - `http://localhost:5173`（Vite dev）
       - `http://localhost:4173`（Vite preview）
     - 其他 Origin 一律拒絕（回 403）

- Q19: 訪客可對哪些 repo 提交 issue / 留言？UI 如何呈現？
  → A: B — **Fetcher 不變 + 在 RepoSummary 新增 `canSubmitIssue: boolean` 欄位 + OverviewPage 兩區塊分流**
     - `scripts/fetch-data.ts` 不新增額外呼叫（不掃 `.github/ISSUE_TEMPLATE/`、不讀 repo settings）
     - 新欄位 `RepoSummary.canSubmitIssue: boolean` 語意：「該 repo 可接受訪客 issue 提交」
       - V1 簡化規則：`milestoneCount > 0 && !isPrivate`（public 且有 milestone = 歡迎外部回饋）
       - 未來可擴充為讀取 `.github/zenbu-milestones.yml` 自訂開關
     - OverviewPage 顯示分成兩區塊：
       - **「接受訪客提交」區塊**：`canSubmitIssue === true` 的 repo，顯示為完整 RepoCard grid（含「建立 issue」按鈕提示）
       - **「無 milestone（僅供瀏覽）」折疊區塊**：`canSubmitIssue === false` 的 repo，延用現有「其他 repos」折疊樣式，不顯示建立 issue 入口
     - RoadmapPage 上的「建立 issue」按鈕僅在 `canSubmitIssue === true` 時可見

## 架構決策摘要（V1 最終版，Q1–Q19 全覽）

| 面向 | 決策 |
|------|------|
| 寫入管道 | 前端 → Cloudflare Worker (with Turnstile) → GitHub REST API |
| 寫入 PAT 位置 | Cloudflare Worker 環境變數（`ZENBU_ORG_WRITE_TOKEN`，`issues:write` + `admin:org` read for types listing） |
| 讀取 PAT 位置 | 維持原樣：GitHub Actions secret（`ZENBU_ORG_READ_TOKEN`，唯讀） |
| Worker 程式碼位置 | 本 repo `worker/` 子目錄（monorepo） |
| Worker 部署方式 | 共用 `build-and-deploy.yml`，透過 Wrangler action 部署到 Cloudflare |
| Worker 部署 Domain | `<worker-name>.workers.dev`（免費子域，V1 不綁 custom domain） |
| Worker CORS 白名單 | `https://zenbuapps.github.io` + `http://localhost:5173` + `http://localhost:4173` |
| 訪客可做 | 建立 issue、留言任何 issue |
| 訪客可提交的 repo 範圍 | `canSubmitIssue === true`（V1 規則：public + milestoneCount > 0） |
| 訪客不可做 | 關閉 issue（保留給 maintainer 用 GitHub 原生 UI）、對 private repo / 無 milestone repo 提交 |
| 反爬蟲 / 反濫用 | Cloudflare Turnstile Managed widget（Modal 內嵌，inline 錯誤顯示） |
| Turnstile Site Key | `VITE_TURNSTILE_SITE_KEY` 環境變數（公開，前端 bundle） |
| Turnstile Secret Key | Worker wrangler secret `TURNSTILE_SECRET_KEY`（僅 Worker 可見） |
| Issue 分類 | **GitHub Issue Types**（org-level，org owner 需預先啟用並定義 Bug/Feature/Task） |
| Issue 預建 label | `待審核`（中文 UTF-8，Worker URL-encode）用於 triage |
| Issue 欄位 | title ≤100 字、body ≤5000 字、type（Issue Type）、body template 含舉報聯絡方式提示 |
| Markdown 編輯器 | @uiw/react-md-editor |
| UI 呈現 | Modal 對話框（max-w-2xl） |
| OverviewPage 分區 | 「接受訪客提交」區塊（`canSubmitIssue=true`）+ 「僅供瀏覽」折疊區塊（`canSubmitIssue=false`） |
| 送出後 UX | 混合方案：樂觀更新本地 state + 下小時 fetcher 重跑同步靜態 JSON |
| 媒體上傳 | V1 不做；編輯器提供教育提示 UI 引導外部上傳 |

## 技術契約影響（Q1–Q19 全覽定稿）

- **`src/data/types.ts`**：
  - `IssueLite` +`type?: string | null`（nullable，舊 issue 可能無 type）
  - `RepoSummary` +`canSubmitIssue: boolean`（V1 規則：public + milestoneCount > 0）
- **`scripts/fetch-data.ts`**：
  - `listIssuesForMilestone` 的 response mapping 讀 `issue.type?.name ?? null`
  - `buildRepoSummary` 計算 `canSubmitIssue = !repo.isPrivate && milestones.length > 0`
- **`public/data/` 新增檔案**：`issue-types.json` — 快取 org-level issue types 清單給前端 select 使用（由 fetcher 透過 `GET /orgs/{org}/issue-types` 產出，需要 `admin:org` read scope，因此改由 Worker 用寫入 PAT 代查或由 read PAT 加 scope 產出，見 open-questions.md）
- **`worker/`**（新 subdir）：Cloudflare Worker TypeScript 程式碼（Turnstile 驗證 + GitHub REST proxy）
- **`.github/workflows/build-and-deploy.yml`**：加 Worker deploy job（wrangler action）
- **新 libs 評估**：
  - `@uiw/react-md-editor` — Markdown 編輯器（前端 bundle）
  - `@cloudflare/workers-types` — Worker TS 型別（dev-only）
  - `wrangler` — Cloudflare CLI（dev-only，CI-only）
  - `@marsidev/react-turnstile` — Turnstile React widget wrapper（可選，亦可直接用官方 script）
- **新 env vars / secrets**：
  - `VITE_TURNSTILE_SITE_KEY`（前端 build-time，公開）
  - `ZENBU_ORG_WRITE_TOKEN`（Worker wrangler secret）
  - `TURNSTILE_SECRET_KEY`（Worker wrangler secret）
  - `CLOUDFLARE_API_TOKEN`（CI repo secret，用於 wrangler deploy）
  - `CLOUDFLARE_ACCOUNT_ID`（CI repo secret，用於 wrangler deploy）
